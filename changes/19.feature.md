Add opt-in automatic config refresh through `SDKConfig.refresh=RefreshConfig(...)`. A daemon thread on `Core` periodically reloads the project config through the existing `Transport`, refreshes the `TrackingQueue`'s `account_id`/`project_id` (atomically alongside the snapshot pointer), and fires the new `LifecycleEvent.CONFIG_UPDATED`. With the default `refresh=None`, no background activity runs.

`Core` gains `refresh_now(wait=False, timeout=5.0)`, `close(flush=True)` (which now flushes pending tracking events through the transport before stopping workers), context-manager support, and a public `refresher_status` property reporting `enabled`, `is_running`, `consecutive_failures`, `last_refresh_at`, `last_success_at`, `last_error_type`, `last_error_at`, `forked_in_child`, and `terminal_failure`.

Failure handling distinguishes transient from terminal. Transient transport failures back off exponentially up to `backoff_max_seconds` and keep retrying; the `on_terminal_failure` callback fires once per failure once the backoff cap is reached. `ConfigValidationError` (bad upstream payload) and apply-callback failures are treated as terminal: the callback fires once and the worker stops, surfaced through `refresh.terminal_failure`.

`RefreshConfig` validates its fields strictly at construction. Misconfigurations (`interval <= 0`, `jitter > interval`, `backoff_initial <= 0`, `backoff_max <= backoff_initial`, `backoff_factor <= 1.0`, etc.) raise `ConfigValidationError` with codes `refresh.invalid_interval`, `refresh.invalid_jitter`, or `refresh.invalid_backoff`.

The worker emits diagnostic events `refresh.start`, `refresh.success`, `refresh.skipped`, `refresh.fail`, `refresh.terminal_failure`, `refresh.fork_detected` (via `os.register_at_fork`), and `refresh.worker_crashed` on `convert_sdk.diagnostics`.
